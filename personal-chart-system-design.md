# 个人榜单系统设计文档

> 状态：设计草案 V1  
> 更新周期：每天 22:00 生成一期榜单  
> 当前主要数据源：网易云音乐用户最近一周听歌排行  
> 后续数据源：实体购买、B 站个人观看记录、其他流媒体平台

## 1. 项目目标

本项目用于生成类似 Billboard 的个人音乐榜单，对用户在不同渠道中的音乐消费和支持行为进行记录、计分与可视化。

系统不把所有指标伪装成真实播放量，而是为每种来源建立独立点数账本，再根据规则生成：

- 滚动七日歌曲榜；
- 月榜；
- 年榜；
- 历史总榜；
- 专辑榜；
- 艺人榜；
- 多数据源综合榜。

第一阶段不接入逐次播放监听。网易云部分基于每天获取的最近一周听歌排行快照，实体与 B 站数据由用户在独立 GUI 中手动录入。

## 2. 核心统计口径

### 2.1 榜单周期

系统每天在本地时间 22:00 更新一次榜单，每次更新形成一个独立统计周期。

例如：

```text
2026-07-17 22:00 至 2026-07-18 21:59:59
```

需要注意：22:00 是本系统保存快照和结算点数的时间，不代表网易云内部也严格按照该时间划分播放记录。

页面应采用如下表述：

> 榜单每日 22:00 更新，网易云数据来自更新时可见的最近一周听歌排行。

不应声称系统获得了 22:00 至次日 22:00 的全部逐次播放记录。

### 2.2 主榜与计分范围

网易云每期获取最近一周听歌排行 Top 100：

- 第 1～50 名：个人主榜 `Personal Chart 50`；
- 第 51～100 名：候补榜 `Bubbling Under 50`；
- 第 1～100 名：全部参与点数发放及长期榜单累计。

因此，一首歌即使从未进入每日 Top 50，只要长期稳定处于第 51～100 名，仍可能凭累计点数进入月榜或年榜。

### 2.3 数据来源分账

各类点数独立保存：

- 网易云流媒体点数；
- 实体点数；
- B 站个人视频点数；
- 其他流媒体点数；
- 历史表现承接奖励；
- 必要时的人工修正。

页面既可以显示综合点数，也必须允许用户查看各来源的点数构成。

## 3. 系统组成

系统分为四个部分：

```text
数据采集与录入
├── 网易云定时采集任务
└── 实体/B站独立录入 GUI

数据存储与计算
├── 原始 JSON 快照
├── SQLite 结构化数据库
├── 点数计算引擎
└── 榜单聚合引擎

可视化展示
└── HTML Web 页面

分享与归档
└── 静态 HTML、JSON、CSV 或图片导出
```

### 3.1 独立 GUI 录入端

实体、B 站及未来需要人工维护的数据不在展示网页中直接编辑，而是通过独立 GUI 录入。

GUI 的职责包括：

- 录入和修改实体购买事件；
- 选择专辑、版本、载体、购买日期和数量；
- 维护实体版本权重；
- 选择实体事件影响的有效曲目；
- 预览实体未来 28 天的点数释放；
- 录入某首歌曲在某日的 B 站观看次数；
- 更正误录数据；
- 管理歌曲、专辑、艺人及不同来源之间的映射；
- 显示计算依据和预计影响，避免黑箱录入。

第一版 GUI 可采用 Python 桌面方案实现，具体框架在开发阶段决定。

### 3.2 HTML 展示端

榜单结果以 HTML Web 页面呈现，负责：

- 查看当前榜单和历史榜单；
- 切换歌曲榜、专辑榜和艺人榜；
- 切换滚动七日榜、月榜、年榜和历史总榜；
- 查看排名、升降、峰值、在榜周期和点数构成；
- 查看歌曲、专辑及艺人的历史趋势；
- 按数据源筛选或比较榜单；
- 导出某一期的静态 HTML 榜单。

HTML 页面只负责查询和展示，不承担主要的数据维护工作。页面的具体视觉布局、组件和交互将在后续单独设计。

## 4. 数据收集

### 4.1 网易云数据

每天 22:00 获取用户最近一周听歌排行 Top 100，并保存：

- 用户标识；
- 采集时间；
- 榜单统计日期；
- 歌曲来源 ID；
- 歌曲名称；
- 艺人；
- 专辑；
- 封面；
- 最近一周播放量；
- 来源排名；
- 来源分数；
- 接口原始字段；
- 采集状态。

接口具有以下边界：

- V1 按网易云数字 UID 匿名读取用户公开可见的最近一周听歌排行，不要求保存账号密码、Cookie 或 Token；
- 该网页数据接口不是面向开发者承诺稳定性的官方开放 API，字段和可用性可能变化；
- 只能观察返回的 Top 100；
- 无法获得完整的逐次播放时间线；
- 排名之外的歌曲可能遗漏；
- 用户隐私设置、接口变化或限流可能导致采集失败；
- 匿名读取被隐私设置阻止时，V1 记为采集失败；本机交互式登录作为后续适配方案，不在服务器保存凭据；
- 网易云数据窗口和更新时间不一定与系统 22:00 周期严格一致。

因此，系统称其为“滚动七日榜数据”，不称为完整日播放记录。

### 4.2 原始快照

每次成功采集后，原始响应以不可变 JSON 文件归档，例如：

```text
data/raw/netease/2026/07/2026-07-17.json
```

原始数据用于：

- 排查接口字段变化；
- 修复解析错误；
- 修改积分公式后重新计算；
- 审计历史榜单的计算依据。

不得只保存最终 Top 50，也不得用新快照覆盖旧快照。

### 4.3 采集失败

采集失败时：

- 保存失败状态及错误信息；
- 按计划进行有限次数重试；
- 不把失败周期视为所有歌曲零分；
- 不使用空数据覆盖上一期快照；
- 在 HTML 页面标记该日数据缺失。

长期榜单在跨时期比较时需要记录有效快照覆盖率。

## 5. 网易云流媒体点数

### 5.1 每日固定点数池

网易云 Top 100 每天共同瓜分固定的流媒体点数池：

```text
每日网易云流媒体点数池：10,000 点
参与范围：Top 100
主榜展示：Top 50
```

固定点数池用于降低用户在不同活跃时期产生的点数通胀，使月份和年份之间更容易比较。

### 5.2 排名权重

V1 排名权重为：

```python
rank_weight = (
    0.08
    + 0.92 * ((101 - rank) / 100) ** 1.35
)
```

设计目的：

- 第 1 名获得明显优势；
- 中段排名平滑递减；
- 第 100 名仍保留基础权重；
- Top 100 中每首歌均可获得点数。

### 5.3 播放强度修正

使用当期 Top 100 最近一周播放量的中位数作为相对基准：

```python
safe_median = max(median_plays, 1)
strength = (weekly_plays / safe_median) ** 0.25
strength = max(0.80, min(strength, 1.25))
```

最终权重：

```python
adjusted_weight = rank_weight * strength
```

播放强度只做有限修正，不应完全取代排名成为点数主导因素。

### 5.4 点数分配

```python
exact_points = (
    10_000
    * adjusted_weight
    / total_adjusted_weight
)
```

整数分配采用最大余数法：

1. 计算每首歌的精确小数点数；
2. 所有歌曲先向下取整；
3. 将剩余点数按小数部分从高到低依次补发；
4. 保证每日最终总点数严格等于 10,000。

### 5.5 综合排名与同分规则

歌曲综合榜不产生并列名次。综合点数相同时，V1 依次比较：

1. 网易云流媒体点数，较高者优先；
2. 实体点数，较高者优先；
3. B 站点数，较高者优先；
4. 上一个已发布同类型榜单中的排名，较高者优先；
5. 稳定内部歌曲 ID，按字典序排序，保证重复计算结果一致。

缺失采集周期不会被视为全员零分；排名变化与最近一个成功发布的同类型榜单比较。

### 5.6 自然周实时榜

系统周榜采用 ISO 自然周（周一至周日），累计本周各个有效日榜周期产生的来源点数：

- 周一至周六显示“统计中”的实时累计结果；
- 周日 22:00 日榜结算后冻结为正式周榜；
- 缺失日不补零，并通过有效日数覆盖率标记数据完整性；
- 周榜的排名变化、峰值和在榜期数只与已发布自然周榜比较，不与同一周内较早的临时结果比较。

### 5.7 月榜和年榜

月榜、年榜不尝试伪造真实月播放量或年播放量，而是累计每日榜单点数：

```text
月度网易云点数 = 当月全部有效周期的网易云流媒体点数之和
年度网易云点数 = 当年全部有效周期的网易云流媒体点数之和
```

跨月份比较时可提供标准月指数：

```python
standard_month_points = (
    earned_points / valid_snapshot_days * 30.4375
)
```

建议覆盖率规则：

- 覆盖率不低于 80%：允许参与标准化跨期比较；
- 覆盖率为 50%～80%：展示实际点数并标记数据不完整；
- 覆盖率低于 50%：不参与历史时期强弱纪录比较。

## 6. 年初历史表现承接奖励

为补偿项目启用前已经被大量播放、但无法还原逐次历史记录的歌曲，可在用户首次建榜或每年年初发放历史表现承接奖励。

奖励依据：

- 网易云累计听歌排行；
- 累计播放量；
- 累计排名档位；
- 当年适用的计分版本。

要求：

- 奖励单独记入 `legacy_bonus`，不伪装成当日流媒体点；
- 页面明确展示奖励来源；
- 奖励规模应设上限，避免老歌长期压制新歌；
- 同一依据不得重复发放；
- 每次发放保存对应的规则版本和原始依据。

具体排名档位、奖励数值和年度衰减方式尚未最终确定，应在正式实现计分引擎前单独完成参数校准。

## 7. 实体点数

### 7.1 基本含义

实体点反映用户通过购买 CD、黑胶、磁带、盒装版本等方式对一张发行作品的支持。

一笔实体购买不会只在录入当天一次性结算，而是开启一个持续 28 天的实体周期。

### 7.2 28 天模拟排名曲线

每个实体事件模拟一首歌在网易云榜单中从较高位置逐渐下降：

```text
第 1 天：参考网易云 #10 点数
……
第 28 天：参考网易云 #100 点数
```

V1 可使用固定排名序列：

```text
10, 12, 15, 18, 21, 24, 27,
30, 33, 36, 39, 42, 45, 48,
51, 54, 57, 60, 64, 68, 72,
76, 80, 84, 88, 92, 96, 100
```

也可以由下式生成，但正式版本只能选择一种方式并通过版本号冻结：

```python
def physical_reference_rank(day: int) -> int:
    progress = (day - 1) / 27
    rank = 10 + 90 * progress**0.85
    return min(round(rank), 100)
```

### 7.3 实体基础公式

实体事件发生时，冻结当日网易云 Top 100 的名次点数曲线。未来 28 天均使用该曲线作为计算基准，避免点数随其他歌曲变化而抖动。

```python
daily_physical_points = (
    frozen_reference_rank_points
    * 10
    * edition_weight
    * purchase_weight
)
```

其中：

- `frozen_reference_rank_points`：实体购买当日模拟名次对应的网易云点数；
- `10`：实体基础倍率；
- `edition_weight`：实体版本权重；
- `purchase_weight`：同批次、多版本或重复购买系数。

实体点数与网易云流媒体点数分账保存。

### 7.4 每首歌曲的实体点

购买一张专辑实体后，专辑中的每首有效歌曲均开启对应的 28 天实体周期，并在当天获得完整的单曲实体点，而不是把一个实体总分平均分给全部歌曲。

有效歌曲默认排除：

- 完全相同录音的重复收录；
- 纯对白或过场；
- 无实质内容的短片段；
- 用户明确标记为不计分的曲目。

Remix、Live、Acoustic 等实质不同录音可以作为独立歌曲。

为避免长专辑在专辑榜和艺人榜中被再次按曲目数放大：

- 歌曲榜：每首有效曲目获得完整实体点；
- 专辑榜：实体事件本身只计算一次事件级点数；
- 艺人榜：实体事件本身只计算一次事件级点数。

这一聚合规则需要在正式实现前通过样例数据验证，确保歌曲榜、专辑榜和艺人榜的量级符合预期。

### 7.5 版本权重

以普通版 CD 为 `1.00` 基准，V1 建议值如下：

| 实体版本 | 权重 |
|---|---:|
| 普通版 CD | 1.00 |
| 豪华版 / 加曲版 CD | 1.15 |
| 限定版 CD | 1.20 |
| 磁带 | 0.90 |
| 标准黑胶 | 1.25 |
| 彩胶 / 限定黑胶 | 1.35 |
| 豪华盒装 / Box Set | 1.50 |
| 单曲 CD | 0.70 |
| 7 英寸黑胶单曲 | 0.85 |

单次实体事件的版本权重建议最高不超过 `1.50`。权重不直接按价格计算。

### 7.6 同一专辑的不同版本

同一专辑不同版本按照购买时间所在的 22:00 榜单周期判断：

- 间隔不足 7 个完整统计周期：视为同一轮购买，后续版本应用版本叠加衰减；
- 间隔达到 7 个完整统计周期：视为新的独立实体事件，重新获得 100% 购买系数并单独开启 28 天周期。

七日内不同版本的建议购买系数：

| 同一轮购买顺序 | 系数 |
|---|---:|
| 第一个版本 | 100% |
| 第二个不同版本 | 60% |
| 第三个不同版本 | 35% |
| 第四个及以后 | 20% |

达到七日间隔后，即使上一实体周期尚未结束，新实体周期也可以开始。重叠日期的点数相加。

示例：

```text
7 月 1 日购买普通 CD
7 月 5 日购买限定黑胶
→ 不足 7 个周期，限定黑胶应用第二版本 60% 系数

7 月 1 日购买普通 CD
7 月 9 日购买限定黑胶
→ 达到 7 个周期，限定黑胶作为独立事件按 100% 计算
```

### 7.7 同一版本重复购买

七日重置规则仅适用于不同版本。同一版本重复购买不因间隔七日恢复完整权重。

建议系数：

| 同版本份数 | 系数 |
|---|---:|
| 第 1 份 | 100% |
| 第 2 份 | 30% |
| 第 3 份 | 15% |
| 第 4 份及以后 | 10% |

### 7.8 豪华版新增曲目

当新版本包含新增曲目时：

- 已存在的共同曲目：应用该版本当前购买系数；
- 新增曲目：可获得该版本完整版本权重；
- 完全相同录音仅更换封面或包装：仍按重复版本处理。

### 7.9 跨月和跨年

实体点按每天实际释放的日期计入对应月份和年份。

例如：

```text
12 月 20 日购买实体
→ 12 月 20 日至 12 月 31 日的释放计入当年
→ 1 月继续释放的部分计入下一年
```

## 8. B 站个人视频点数

### 8.1 第一阶段录入方式

B 站数据暂不自动接入接口，由用户在独立 GUI 中手动录入：

- 日期；
- 关联歌曲；
- 当日观看次数；
- 可选备注；
- 可选视频标题或 BV 号，仅用于记录来源。

同一首歌的不同视频版本按歌曲汇总：

```text
官方 MV 3 次 + Live 2 次 + 二创 1 次
= 该歌曲当日观看 6 次
```

B 站点数表示用户个人观看行为，不表示 B 站全站市场热度。

### 8.2 数量级分档

V1 建议采用分档，而不是观看次数与点数无限线性增长：

| 当日观看次数 | B 站点数 |
|---:|---:|
| 0 | 0 |
| 1 | 20 |
| 2 | 35 |
| 3～4 | 55 |
| 5～7 | 80 |
| 8～12 | 110 |
| 13～20 | 145 |
| 21～35 | 180 |
| 36～60 | 215 |
| 61 次及以上 | 250 |

设计目的：

- 一次观看也能留下贡献；
- 多次观看有明显增益；
- 极高观看次数不会无限放大；
- 单首歌单日 B 站个人视频点最高为 250 点。

该分档属于初始平衡参数，正式使用前应与网易云第 50 名附近的日均点数、实体周期点数进行样例校准。

### 8.3 专辑榜和艺人榜

B 站点数首先属于歌曲，再向上聚合：

```python
album_bilibili_points = sum(song_bilibili_points)
```

合作歌曲在艺人榜中的分配方式暂定为平均分配：

```python
artist_bilibili_points = (
    song_bilibili_points / credited_artist_count
)
```

如果后续改为主艺人与客席艺人不同权重，需要增加新的计分版本，不追溯修改已发布历史榜单。

## 9. 专辑榜与艺人榜

### 9.1 网易云部分

专辑榜和艺人榜使用每日 Top 100 的全部歌曲点数，而不是只使用展示的 Top 50。

专辑网易云点数：

```python
album_points = sum(
    song.netease_points
    for song in album_chart_entries
)
```

艺人合作歌曲建议平均分配：

```python
artist_points = (
    song.netease_points / credited_artist_count
)
```

### 9.2 多来源构成

每个专辑和艺人的详情应分别显示：

- 网易云流媒体点；
- 实体点；
- B 站个人视频点；
- 其他来源点数；
- 综合点数。

不同来源不能只保留合计值，否则无法解释综合排名变化。

## 10. 综合榜

综合榜由各来源点数组合而成，但原始来源账本始终保留。

概念公式：

```python
combined_points = (
    netease_streaming_points
    + physical_points
    + bilibili_points
    + other_source_points
    + legacy_bonus
)
```

是否需要对不同来源再应用综合权重，尚未最终确定。正式实现综合榜前需要使用模拟数据验证：

- 网易云长期表现是否仍是基础；
- 实体购买是否具有足够影响但不会永久压榜；
- B 站手动观看是否有存在感但不会因高频录入失控；
- 新旧歌曲是否能公平竞争；
- 跨年度规则是否可解释。

## 11. 数据存储设计

### 11.1 存储策略

第一版采用：

```text
原始数据：不可变 JSON 文件
结构化数据：SQLite
展示：动态 HTML Web 页面
分享：静态 HTML 快照
```

SQLite 适合本地个人项目，无须额外部署数据库服务；未来若改为多用户在线服务，可迁移到 PostgreSQL。

## 12. 计分版本与历史稳定性

每条快照、事件和账本记录必须保存计分版本，例如：

```text
netease-streaming-v1
physical-v1
bilibili-manual-v1
legacy-bonus-v1
```

规则调整后创建新版本，不直接改变已经发布的历史榜单。若用户主动选择重算历史，系统应：

- 使用原始快照和事件重算；
- 记录新的重算版本；
- 保留旧版本或生成备份；
- 明确显示榜单采用的规则版本。

跨版本比较优先使用点数占同期可发点数的比例，而不是简单设置不透明的年代系数。

## 13. GUI 与 HTML 的职责边界

### 13.1 GUI 负责写入

GUI 是数据维护入口：

- 创建、编辑、撤销人工事件；
- 管理实体版本及曲目映射；
- 录入 B 站每日观看次数；
- 查看计算预览；
- 执行必要的人工修正；
- 提示冲突、重复录入和七日版本规则。

### 13.2 HTML 负责读取和展示

HTML Web 页面默认只读：

- 展示当前及历史榜单；
- 查询歌曲、专辑和艺人详情；
- 展示趋势、纪录和点数来源；
- 切换数据源和时间范围；
- 导出榜单。

这种分离可以降低误操作风险，也使展示网页保持清爽。

## 15. 当前已确定的 V1 摘要

- 每天 22:00 生成一期榜单；
- 网易云按数字 UID 读取公开可见的最近一周排行 Top 100；
- Top 50 用于主榜展示，Top 100 全部参与计分；
- 网易云 Top 100 每天共同瓜分固定 10,000 点；
- 月榜和年榜累计榜单点数，不冒充真实播放量；
- 实体购买为专辑内每首有效歌曲开启 28 天点数周期；
- 实体每天参考一条从网易云 #10 下降到 #100 的模拟排名曲线，并按参考点数的 10 倍计算；
- 实体版本拥有不同权重；
- 同一专辑不同版本间隔达到 7 个完整周期时，分别作为独立实体事件；
- 同一版本重复购买不适用七日重置；
- B 站由用户手动录入每首歌曲每日观看次数，并按数量级分档计分；
- 实体和 B 站数据使用独立 GUI 维护；
- 榜单使用 HTML Web 页面展示；
- 原始快照保存为 JSON，结构化数据和点数账本保存于 SQLite；
- 所有规则均通过计分版本冻结，保证历史榜单可解释和可追溯。
